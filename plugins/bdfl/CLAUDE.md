# BDFL Plugin — Maintainer Guide

## Architecture Overview

BDFL (Benevolent Dictator For Life) is a skill-only plugin with no hooks, commands, or TypeScript. It provides two user-invokable skills that share a single architecture reference document, plus a migration milestones document that defines the ordered path from any project to BDFL compliance.

**Three concerns, three homes:**

| Concern | File | Description |
|---------|------|-------------|
| What BDFL means | `architecture.md` | Normative target — the done state |
| How to get there | `milestones.md` | Ordered phases with concrete done criteria |
| Where a repo is | `.bdfl.yaml` | State ledger in the target repo (generated by migrate) |

## Directory Structure

```
plugins/bdfl/
├── .claude-plugin/
│   └── plugin.json
├── CLAUDE.md                         # This file (maintainer guide)
├── README.md                         # User quick start
└── skills/
    ├── init/
    │   ├── SKILL.md                  # Bootstrap skill
    │   └── references/
    │       └── architecture.md       # Canonical architecture spec (single source of truth)
    └── migrate/
        ├── SKILL.md                  # Migration skill
        └── references/
            ├── architecture.md       # Symlink → ../../init/references/architecture.md
            └── milestones.md         # Ordered migration phases with done criteria
```

## Component Responsibilities

### `architecture.md`

The single source of truth for what a BDFL-certified project looks like: runtime, package manager, TypeScript config, Effect.ts, component frameworks, tooling, hooks, containerisation, NixOS. Shared by both skills via symlink. Describes the **target state**, not implementation steps.

### `milestones.md`

Defines the ordered migration phases and their concrete done criteria. Only used by the migrate skill. Phases must be completed in order. Each phase has verifiable exit criteria.

**Phase order:** nix (optional) → specs → formatter → linter → typeChecker → build → catchAllHooks → monorepo → architecture → refinedHooks → containerisation → nixReadiness

### `skills/init/SKILL.md`

Scaffolds a new project from scratch that meets the full BDFL architecture from day one. Takes `$ARGUMENTS` as a project outline. No `.bdfl.yaml` generated — the project starts at full compliance.

### `skills/migrate/SKILL.md`

Analyses an existing project, reads or generates `.bdfl.yaml`, finds the next incomplete phase, analyses the gap against done criteria, and reports. Does **not** execute migration work or prescribe how to break it down — that is the user's choice. Designed to be run repeatedly — each invocation checks one phase.

### `.bdfl.yaml` (generated in target repos)

A state ledger tracking migration progress. Not a config file — contains no policies or tool preferences. Statuses: `pending | in-progress | done | skipped`.

## How It Works

1. User invokes `/bdfl:init <outline>` or `/bdfl:migrate`
2. Claude loads the SKILL.md which references architecture.md (and milestones.md for migrate)
3. **Init**: scaffolds a complete project matching the architecture spec
4. **Migrate**: reads `.bdfl.yaml`, finds next phase, analyses gap, reports
5. User implements the phase using their preferred workflow
6. User re-runs `/bdfl:migrate` to verify done criteria and advance

## Key Design Principles

- **The plugin is the opinion**: no configurability, no profiles, no policy knobs. BDFL dictates the strictest everything.
- **The plugin is not an execution engine**: it defines milestones and done criteria. It does not own how work is broken down or implemented.
- **`.bdfl.yaml` is a state ledger, not config**: it records where a project is, not where it should go. The architecture and milestones documents are the authority.
- **Phases are ordered for safety**: specs before tooling changes (safety net), tooling before hooks (something to hook into), hooks before structural changes (prevent regressions), structure before architecture migration (stable foundation).

## Common Maintenance Tasks

### Updating the architecture

Edit `skills/init/references/architecture.md` only. The migrate skill picks up changes automatically via the symlink.

### Updating migration phases

Edit `skills/migrate/references/milestones.md`. Phase ordering changes should be carefully considered — later phases depend on earlier ones.

### Adding a new skill

1. Create `skills/<name>/SKILL.md`
2. If it needs the architecture reference, symlink it: `ln -s ../../init/references/architecture.md skills/<name>/references/architecture.md`
3. Update this CLAUDE.md

## Common Pitfalls

- Do not duplicate `architecture.md` — always maintain the symlink
- The init skill expects `$ARGUMENTS` — invoking without a project outline produces a generic scaffold
- The migrate skill analyses and reports only — it must not execute migration work
- Milestones have concrete done criteria for a reason — do not mark phases done without verifying them
- `.bdfl.yaml` belongs in the target repo, not in this plugin
