# BDFL Specification

**Status:** Implemented
**Last Updated:** 2026-04-02

## 1. Overview

### Purpose

Opinionated project architecture plugin for Claude Code. Provides two skills — bootstrap a new project from scratch at full BDFL compliance (`init`), or incrementally migrate an existing project through ordered milestones toward BDFL certification (`migrate`). The plugin codifies tooling and architecture decisions so they are applied consistently without re-deciding each time.

### Goals

- Define a single canonical project architecture (runtime, language config, libraries, tooling, hooks, containerisation)
- Scaffold compliant projects from scratch via `bdfl:init`
- Guide incremental migration via ordered milestones with verifiable done criteria
- Track migration progress in a state ledger (`.bdfl.yaml`) so work can be resumed across sessions

### Non-Goals

- Not an execution engine — defines targets and milestones, does not implement migration work
- No configurability, profiles, or policy knobs — the plugin is the opinion
- Does not manage cross-project coordination or fleet-wide migration tracking
- `.bdfl.yaml` is not a config file — it records state, not policy

## 2. Architecture

Skill-only plugin with no hooks, commands, TypeScript, or agents. Two skills share one architecture reference via symlink.

```
plugins/bdfl/
├── .claude-plugin/plugin.json
├── skills/
│   ├── init/
│   │   ├── SKILL.md                           # Bootstrap skill
│   │   └── references/
│   │       └── architecture.md                # Canonical target state (single source of truth)
│   └── migrate/
│       ├── SKILL.md                           # Migration skill
│       └── references/
│           ├── architecture.md → (symlink)    # Points to init/references/architecture.md
│           └── milestones.md                  # Ordered migration phases with done criteria
```

### Component Relationships

**Three concerns, three homes:**

| Concern | File | Description |
|---------|------|-------------|
| What BDFL means | `architecture.md` | Normative target — the done state |
| How to get there | `milestones.md` | Ordered phases with concrete done criteria |
| Where a repo is | `.bdfl.yaml` | State ledger in the target repo (generated by migrate) |

### Skills

| Skill | Input | Output | Behaviour |
|---|---|---|---|
| `bdfl:init` | `$ARGUMENTS` (project outline) | Scaffolded project | Reads architecture.md, scaffolds a complete project matching every requirement. No `.bdfl.yaml` — project starts at full compliance. |
| `bdfl:migrate` | Current project state | Updated `.bdfl.yaml` + gap report | Reads architecture.md + milestones.md. Walks phases in order, finds next incomplete one, analyses gap against done criteria, reports. Does not execute work. |

### Reference Documents

| Reference | Used By | Purpose |
|---|---|---|
| `skills/init/references/architecture.md` | Both skills | Canonical target state for a BDFL-certified project |
| `skills/migrate/references/milestones.md` | `migrate` only | Ordered migration phases with verifiable done criteria |

The architecture reference is shared via symlink (`migrate/references/architecture.md → ../../init/references/architecture.md`). Only the init copy is edited; migrate picks up changes automatically.

## 3. Data Model

### architecture.md (Target State)

Defines the full technology stack for a compliant project:

- **Runtime**: Node.js (LTS) + pnpm + Volta version pinning
- **Language**: TypeScript with strictest config (strict, noUncheckedIndexedAccess, exactOptionalPropertyTypes, etc.)
- **Core library**: Effect.ts ecosystem for error handling, concurrency, DI, schema, HTTP, config
- **Components**: React (web), Expo/React Native (mobile), `@effect/platform` HTTP (server)
- **Build/test**: Vite + Vitest (exception: jest + jest-expo for Expo)
- **Tooling**: `@typescript/native-preview` (tsgo), ESLint + typescript-eslint (flat config), oxfmt
- **Hooks**: PostToolUse format+lint per-file, Stop/SubagentStop verify, pre-commit verify
- **Containerisation**: Dockerfile + `dev.sh` via Podman
- **NixOS**: `flake.nix` dev shell
- **Monorepo**: pnpm workspaces + shared tsconfig.base.json + project references (if applicable)

### milestones.md (Migration Phases)

Twelve ordered phases. Each has concrete done criteria. Phases must be completed in order — later phases depend on earlier ones being stable.

| # | Phase | Purpose |
|---|---|---|
| 0 | `nix` | NixOS dev shell (optional — can defer to `nixReadiness`) |
| 1 | `specs` | Behavioural safety net: docs + tests using incumbent tooling |
| 2 | `formatter` | Install oxfmt, bulk format, `.git-blame-ignore-revs` |
| 3 | `linter` | ESLint + typescript-eslint, zero violations |
| 4 | `typeChecker` | Strictest TypeScript via tsgo, zero errors |
| 5 | `build` | Vite + Vitest, migrate tests, `pnpm verify` works |
| 6 | `catchAllHooks` | Blunt hooks: Stop/SubagentStop + pre-commit run `pnpm verify` |
| 7 | `monorepo` | pnpm workspaces + project references (skip if single package) |
| 8 | `architecture` | Migrate core logic to Effect.ts |
| 9 | `refinedHooks` | Add per-file PostToolUse format+lint hooks |
| 10 | `containerisation` | Dockerfile + `dev.sh` via Podman |
| N-1 | `nixReadiness` | Final NixOS gate (if `nix` was skipped at phase 0) |

**Phase ordering rationale:** specs before tooling (safety net), tooling before hooks (something to hook into), hooks before structure (prevent regressions), structure before architecture (stable foundation).

### .bdfl.yaml (State Ledger)

Generated and maintained by `bdfl:migrate` in the target project root. Tracks per-phase status.

```yaml
phases:
  nix: skipped
  specs: done
  formatter: done
  linter: in-progress
  typeChecker: pending
  # ...
```

**Statuses:** `pending` | `in-progress` | `done` | `skipped`

Not a config file — contains no policies or tool preferences. The architecture and milestones documents are the authority.

## 4. Interfaces

### User-Facing Skills

- **`/bdfl:init <outline>`** — scaffolds a new project. Takes a brief description of the project, its purpose, and which components are needed (web, mobile, server). Produces a fully compliant project directory.

- **`/bdfl:migrate`** — analyses migration progress. On first run: walks all phases, generates `.bdfl.yaml` with early done phases and remaining pending. On subsequent runs: finds first non-done phase, analyses gap, reports. On re-run after implementation: verifies done criteria, advances if met. Reports only — never executes migration work.

### Migrate Workflow

```
First run → generate .bdfl.yaml → report first pending phase
  ↓
User implements phase externally
  ↓
Re-run → verify done criteria → advance to next phase
  ↓
Repeat until all phases done/skipped → BDFL certified
```

## 5. Design Decisions

- **Decision**: The plugin is the opinion — no configurability.
  **Rationale**: BDFL dictates the strictest everything. Configurability would undermine the point. If the architecture changes, the reference document changes for everyone.

- **Decision**: Migrate analyses and reports only — never executes work.
  **Rationale**: How work is broken down and implemented is the user's choice. The plugin defines what and in what order, not how. This keeps the plugin simple and avoids dangerous automated refactoring.

- **Decision**: `.bdfl.yaml` is a state ledger, not config.
  **Rationale**: Separates "where are we" from "where should we go". The architecture and milestones documents are the authority. The ledger just tracks progress.

- **Decision**: Phases are strictly ordered.
  **Rationale**: Later phases depend on earlier ones. Specs provide a safety net before tooling changes. Tooling changes give hooks something to hook into. Hooks prevent regressions during structural changes.

- **Decision**: Architecture shared via symlink, not duplication.
  **Rationale**: Single source of truth. Edit once, both skills see the update. Eliminates drift between init and migrate targets.

- **Decision**: First-run phase walking marks later phases as `pending` even if they appear satisfied.
  **Rationale**: A later phase hasn't been verified in the context of a stable migration chain. Early phases may change things that invalidate later assumptions.

## 6. Testing

No automated test suite. The plugin is pure markdown with no runtime code. Verification is behavioural — the skills are tested by using them on real projects. The migrate skill's done criteria serve as built-in verification for each phase.
